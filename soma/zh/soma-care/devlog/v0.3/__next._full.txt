18:T1747,<h1 id="v03--从技术仪表盘变成产品界面">V0.3 — 从"技术仪表盘"变成"产品界面"</h1>
<p>v0.2 以"技术跑通"为导向，UI 是 Anima 仪表盘 + 仿真窗口的四列布局。v0.3 的目标是把信息层级反过来：<strong>让访客第一眼看到的是产品（护理机器人在病房里做事），而不是技术架构</strong>。Anima 的 L0–L5 细节退居可折叠抽屉，供想看技术的人主动展开。</p>
<h2 id="ui-从暗色技术风改为浅色临床风">UI 从暗色技术风改为浅色临床风</h2>
<ul>
<li>主题从 <code>zinc-950</code> 暗色改为白底 + <code>zinc-200</code> 边框，更像真实临床产品的界面</li>
<li>新增 <code>bci/</code> 组件组：<code>SessionHeader</code> / <code>ChannelHealth</code> / <code>NeuralActivity</code> / <code>DecoderOutput</code> / <code>FiringRateBars</code> —— 都标注 decorative，和 v0.1 定下的"不伪装神经解码"诚信线一致</li>
<li>新增 <code>shell/</code> 组件组：<code>MissionRibbon</code>（顶部任务条）/ <code>BciDrawer</code>（左抽屉）/ <code>TaskDrawer</code>（右抽屉）—— 抽屉默认收起，把"产品视图"让给 MJPEG 主画面</li>
<li><code>SimulationView</code> 升级为多相机 + E-stop + 状态轮询的产品级组件</li>
</ul>
<h2 id="后端五路相机渲染管线">后端：五路相机渲染管线</h2>
<p>从单路 <code>demo_view</code> 扩到五路：<code>demo_view</code> / <code>grasp_view</code> / <code>bedside_view</code> / <code>top_down</code> / <code>tv_view</code>。</p>
<p>技术细节：</p>
<ul>
<li><code>_CameraFeed</code> 类，每路一个 <code>mujoco.Renderer</code> + 自己的 latest_jpeg + condvar</li>
<li>单个渲染线程从一份共享 <code>MjData</code> 同步刷所有 feed</li>
<li><code>/api/sim/cameras</code> 路由广告相机列表，<code>/api/sim/mjpeg?camera=X</code> 按名取流</li>
<li>关掉每个 renderer 的 <code>mjVIS_RANGEFINDER</code> 标志（否则地面会被黄色测距光束糊满）</li>
</ul>
<h2 id="e-stop-旁路通道">E-stop 旁路通道</h2>
<p>第一条<strong>独立于 LLM / 行为树</strong>的紧急停止通道：</p>
<ul>
<li><code>SimManager.estop_active</code> 用 <code>threading.Event</code> 暴露全局 flag</li>
<li><code>POST /api/sim/estop</code> 设 flag 并把底盘速度归零；<code>POST /api/sim/estop/clear</code> 清掉</li>
<li><code>SimSkillBehaviour.update</code> 每 tick 第一件事检查 flag → 立即 FAILURE，不走 LLM / BT 重建</li>
<li><code>Reset Sim</code> 同时清 E-stop flag</li>
<li>前端右上角红色按钮；激活时变 amber "Clear E-stop"，顶部出现红色警示条</li>
</ul>
<p>这是后来 v0.4 规划"三种非信号通路旁路"的第一条原型。</p>
<h2 id="六个意图全部跑通">六个意图全部跑通</h2>
<p>每条都给出可演示动作：</p>
<ul>
<li><code>DRINK_WATER</code> — 五步抓杯送床边（沿用 v0.2）</li>
<li><code>ADJUST_TV</code> — TV 屏的 emission 材质变亮蓝</li>
<li><code>CALL_HELP</code> — 机器人抬臂做信标 + 护士 mocap 出现在病床旁停留 6 秒后归位，同时 WebSocket 推 <code>help.called</code> 事件</li>
<li><code>GOTO_BED</code> — base 驱到床头位 <code>(4.5, 1.3, -π/2)</code></li>
<li><code>TURN_OFF_LIGHT</code> / <code>TURN_ON_LIGHT</code> — 五盏光源 diffuse 清零或恢复 + 吊灯发光材料置黑或亮。<strong>关键细节</strong>：subtask 名 <code>turn_off_light</code> / <code>turn_on_light</code> 决定目标态，不是纯 toggle，避免"再说一次关灯反而开了灯"的语义错误</li>
<li>E-stop 中点 → outcome=cancel</li>
</ul>
<h2 id="踩过的坑值得写下来">踩过的坑（值得写下来）</h2>
<p><strong>床头位避障</strong>：最早把 <code>BEDSIDE_POSE</code> 设 <code>(4.2, 1.0, -π/2)</code>，unicycle PID 在 <code>(3.7, 1.0)</code> 附近卡住反复震荡 50+ 秒到超时。根因是 nightstand 碰撞箱的 y 边到 <code>0.79</code>，Stretch 底盘半径约 <code>0.35</code>，机器人中心 y &#x3C; <code>1.14</code> 就会卡在 nightstand 北缘。改到 <code>y=1.3</code> 留 <code>0.16m</code> 安全余量，一次到位。</p>
<p><strong>渲染性能天花板</strong>：Hetzner CPX31 没 GPU，osmesa 软件渲染五路相机满载 ~0.4 fps。直接影响是任何由 BT 驱动的"滑入式"动画（比如最初设计的"护士从门口走入"）在渲染流上只采样到 1–2 帧，视觉上等同于瞬移。解决方式是把动画改成"目标态保持 ≥ 渲染周期"的离散关键帧——CallHelpSkill 的 6 秒停留就是为此设的。这条硬约束<strong>直接促成了 v0.4 放弃 Hetzner live demo，改为本地 Mac Metal 渲染 + 离线视频交付</strong>。</p>
<h2 id="canonical-subtask-safety-net">Canonical Subtask Safety Net</h2>
<p><code>l1_parser.py</code> 加了一份 <code>_CANONICAL_PLANS</code>：每个 intent 都有一份确定性的 BT 兜底计划。LLM parse 失败也能跑——演示场合下要的是"永远别卡住"，而不是"永远用最优路径"。</p>

<li><code>/api/sim/cameras</code> 返回五个相机</li>

<li>六个意图在前端逐一跑通 outcome=success，E-stop outcome=cancel</li>
</ul>
