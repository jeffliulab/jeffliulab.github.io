4:T136e,<h1 id="v02--把-mock-skills-换成真正自主的-stretch">V0.2 — 把 mock skills 换成真正自主的 Stretch</h1>
<p>v0.1 证明了 ANIMA 认知栈在语言输入下能跑通；v0.2 要把底下的 mock skills 换成<strong>真自主执行</strong>的机器人——每次运行都从 MJCF 查物体位姿、算 IK、步进物理，不是脚本重放。</p>
<p>选型上走了一次 pivot，值得记录。</p>
<h2 id="pivot从-ros-2--gazebo-改到纯-python-mujoco">Pivot：从 ROS 2 + Gazebo 改到纯 Python MuJoCo</h2>
<p><strong>原方案</strong>是 Docker + ROS 2 Humble + Gazebo Harmonic + MoveIt 2 + Nav2——标准"重型"机器人栈。</p>
<p>执行时发现三个问题：</p>
<ol>
<li><code>hello-robot</code> 官方不提供 <code>stretch_moveit2</code> 的 apt 包或 GitHub 配置，要自己写 SRDF + kinematics config（3–5 小时工作量）</li>
<li><code>stretch_mujoco</code> 的 Python API 本身已经够用：<code>sim.pull_camera_data()</code> → RGB 帧；<code>sim.move_to(Actuators.lift, pos)</code> + <code>wait_until_at_setpoint</code> → 关节位置控制；<code>sim.set_base_velocity(v, ω)</code> → 底盘速度。Stretch 手臂只有 5 DoF（lift Z + arm 伸缩 + wrist yaw/pitch/roll），分析 IK 10 行代码内能搞定</li>
<li>Nav2 对封闭的单房间 + 走廊场景明显过重</li>
</ol>
<p><strong>新方案</strong>：纯 Python <code>stretch_mujoco</code> API + 分析 IK + unicycle PID。ANIMA 的 L3 skill 直接调 <code>stretch_mujoco</code>，不启动任何 ROS 节点。</p>
<p>这个选择顺带给了几个好处：</p>
<ul>
<li>MuJoCo 原生支持 macOS，开发机也能跑，不再被 Linux 云主机的 VSCode Remote SSH 绑死</li>
<li>部署简化到三个进程（FastAPI + Next.js + MuJoCo），不再需要 rosdep / colcon / DDS 调参</li>
<li>真机迁移路径不被堵死：v1.x 想上 ROS 只需把 <code>SimSkillBehaviour</code> 换成 <code>Ros2SkillBehaviour</code>，上层 <code>py_trees</code> 编排零改</li>
</ul>
<p>原本的 ROS 2 Humble + stretch_ros2 workspace 保留不删，当作未来接真机的参考。</p>
<h2 id="l3-技能层落地六个原语">L3 技能层落地六个原语</h2>
<p>新增 <code>SimSkillBehaviour</code> 基类 + 六个具体 skill，共享一个 blackboard dict：</p>
<ul>
<li><code>locate</code> — 在仿真里查目标 body 位姿</li>
<li><code>navigate</code> — unicycle PID，带 align / approach 两段 hysteresis 避震荡</li>
<li><code>grasp</code> — 分析 IK + 闭合爪</li>
<li><code>lift</code> — 抬起到安全高度</li>
<li><code>deliver</code> — 导到床边位</li>
<li><code>release</code> — 打开爪子</li>
</ul>
<p>同时在 L2 Planner 里加了"仿真不可用时自动回落到 v0.1 mock skill"的兜底，保证本地零配置也能跑。</p>
<h2 id="仿真视图进前端">仿真视图进前端</h2>
<p>后端新增 <code>sim/manager.py</code> 持有 <code>StretchMujocoSimulator</code>，并自己渲染 <code>demo_view</code> 第三人称相机。<code>stretch_mujoco</code> 的 camera pipeline 不暴露自定义 MJCF 相机，所以绕过它直接用 <code>mujoco.Renderer</code>。</p>
<p>三条路由：</p>
<ul>
<li><code>/api/sim/mjpeg</code> — multipart MJPEG 流</li>
<li><code>/api/sim/reset</code> — 场景归位</li>
<li><code>/api/sim/status</code> — 状态探针</li>
</ul>
<p>前端加一个 <code>SimulationView.tsx</code>（<code>&#x3C;img></code> 接 MJPEG + Reset 按钮），整体布局从三列扩到四列（nav / sim live / intent+BT / factors）。</p>
<h2 id="l5-数据开始驱动评估">L5 数据开始驱动评估</h2>
<p><code>l5_assessment.py</code> 里，<code>p_skill</code> 从固定 0.91 改成"从 <code>pea_log.jsonl</code> 读滑窗真实成功率"——PEA 终于开始喂回 GOA 计算，不再是静态常数。</p>
