4:T1a5a,<h1 id="v03--from-tech-dashboard-to-product-interface">V0.3 — From "tech dashboard" to "product interface"</h1>
<p>v0.2 was optimized for "does the tech work," with a four-column layout of ANIMA dashboards + a simulation window. v0.3 inverts the information hierarchy: <strong>first-time viewers should see the product (a care robot doing things in a ward), not the tech stack</strong>. ANIMA's L0–L5 details move into collapsible drawers — available on demand for anyone who wants to dig in.</p>
<h2 id="ui-from-dark-tech-theme-to-light-clinical-theme">UI: from dark tech theme to light clinical theme</h2>
<ul>
<li>Theme shifts from <code>zinc-950</code> dark to white + <code>zinc-200</code> borders, matching the aesthetic of a real clinical product surface</li>
<li>New <code>bci/</code> component group: <code>SessionHeader</code> / <code>ChannelHealth</code> / <code>NeuralActivity</code> / <code>DecoderOutput</code> / <code>FiringRateBars</code> — all labeled decorative, consistent with the honesty line from v0.1 ("do not fake neural decoding")</li>
<li>New <code>shell/</code> component group: <code>MissionRibbon</code> (top mission bar) / <code>BciDrawer</code> (left drawer) / <code>TaskDrawer</code> (right drawer) — drawers collapse by default so the MJPEG feed owns the visual field</li>
<li><code>SimulationView</code> upgrades to a product-grade component with multi-camera switching, E-stop, and status polling</li>
</ul>
<h2 id="backend-five-camera-render-pipeline">Backend: five-camera render pipeline</h2>
<p>From a single <code>demo_view</code> to five: <code>demo_view</code> / <code>grasp_view</code> / <code>bedside_view</code> / <code>top_down</code> / <code>tv_view</code>.</p>
<p>Implementation detail:</p>
<ul>
<li><code>_CameraFeed</code> class with one <code>mujoco.Renderer</code> per feed plus its own latest_jpeg and condvar</li>
<li>A single render thread syncs all feeds from one shared <code>MjData</code></li>
<li><code>/api/sim/cameras</code> advertises the list; <code>/api/sim/mjpeg?camera=X</code> picks one by name</li>
<li>Disable each renderer's <code>mjVIS_RANGEFINDER</code> flag (otherwise the floor fills with yellow rangefinder beams)</li>
</ul>
<h2 id="e-stop-bypass-channel">E-stop bypass channel</h2>
<p>The first emergency-stop path <strong>independent of the LLM and behavior tree</strong>:</p>
<ul>
<li><code>SimManager.estop_active</code> exposes a global <code>threading.Event</code></li>
<li><code>POST /api/sim/estop</code> sets the flag and zeros base velocity; <code>POST /api/sim/estop/clear</code> lifts it</li>
<li><code>SimSkillBehaviour.update</code> checks the flag as its first action each tick → immediate FAILURE, bypassing LLM / BT rebuild</li>
<li><code>Reset Sim</code> also clears E-stop</li>
<li>Frontend: red button top-right; when active it becomes an amber "Clear E-stop", with a red warning band across the top</li>
</ul>
<p>This is the prototype of what v0.4 formalizes as "three non-signal-path bypass channels."</p>
<h2 id="all-six-intents-running-end-to-end">All six intents running end-to-end</h2>
<p>Each intent has a concrete demonstrable action:</p>
<ul>
<li><code>DRINK_WATER</code> — five-step grasp-and-deliver (inherited from v0.2)</li>
<li><code>ADJUST_TV</code> — TV screen emission material brightens to sky blue</li>
<li><code>CALL_HELP</code> — robot arm raises as a beacon; a nurse mocap appears at bedside and holds for 6 s before retreating; a <code>help.called</code> event is pushed over WebSocket</li>
<li><code>GOTO_BED</code> — base drives to the bedside head pose <code>(4.5, 1.3, -π/2)</code></li>
<li><code>TURN_OFF_LIGHT</code> / <code>TURN_ON_LIGHT</code> — five light sources zero or restore diffuse; ceiling lamp emission toggles dark / bright. <strong>Key detail</strong>: the subtask name (<code>turn_off_light</code> / <code>turn_on_light</code>) determines the target state, not a pure toggle — otherwise "say 'turn off light' twice" would surprise the user by turning the light back on</li>
<li>E-stop mid-run → outcome=cancel</li>
</ul>
<h2 id="lessons-worth-recording">Lessons worth recording</h2>
<p><strong>Bedside-pose obstacle avoidance</strong>: the first choice of <code>BEDSIDE_POSE</code> was <code>(4.2, 1.0, -π/2)</code>. The unicycle PID got stuck around <code>(3.7, 1.0)</code> oscillating for 50+ seconds until timeout. Root cause: the nightstand's collision box has a y-edge at <code>0.79</code>; the Stretch base radius is ~<code>0.35</code>, so any robot center with y &#x3C; <code>1.14</code> collides with the nightstand's north edge. Changing to <code>y=1.3</code> leaves a <code>0.16 m</code> safety margin, and the robot reaches the pose on the first try.</p>
<p><strong>Rendering ceiling</strong>: Hetzner CPX31 has no GPU. Software rendering five cameras via osmesa tops out at ~0.4 fps. The direct consequence: any BT-driven "slide-in" animation (like the original design where the nurse walks in through the door) only samples 1–2 frames and visually reads as teleportation. The fix is to rewrite animations as discrete keyframes that hold each target state ≥ one render period — the 6-second hold in <code>CallHelpSkill</code> exists for exactly this reason. This hard constraint <strong>directly drove v0.4's decision to abandon the Hetzner live demo and move to local Mac Metal rendering + offline video delivery</strong>.</p>
<h2 id="canonical-subtask-safety-net">Canonical subtask safety net</h2>
<p><code>l1_parser.py</code> gains a <code>_CANONICAL_PLANS</code> table: every intent carries a deterministic BT fallback plan. Even if the LLM parse fails, the demo doesn't stall. In a demonstration setting, "never get stuck" beats "always take the optimal path."</p>
<h2 id="acceptance-passed">Acceptance (passed)</h2>
<ul>
<li><code>/api/sim/cameras</code> returns five cameras</li>
<li><code>tv_view</code> MJPEG: 200 OK, ~50 KB per frame</li>
<li>Two consecutive <code>/api/sim/reset</code> calls no longer SEGV (an earlier version had a lifetime race between the render thread and <code>_model</code> — fix lives in <code>SimManager.stop()</code>)</li>
<li>All six intents run <code>outcome=success</code> end-to-end in the frontend; E-stop gives <code>outcome=cancel</code></li>
</ul>

Delivery was still the Hetzner live demo. v0.4 onwards happens in the standalone <code>jeffliulab/soma-care</code> repo and shifts toward local high-frame-rate rendering, offline video delivery, and finer contact-rich operations.
